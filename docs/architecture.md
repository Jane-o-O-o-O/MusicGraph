# MusicGraph 架构方案

## 1. 目标

MusicGraph 第一阶段只解决两个问题：

1. 用户可以搜索音乐实体
2. 用户可以用 3D 图谱浏览实体关系

当前不纳入第一阶段的内容：

- GraphRAG
- 向量检索
- 自动抽取流水线
- 复杂权限系统
- MySQL 与 Neo4j 双写同步

## 2. 总体架构

推荐采用三层结构：

1. 数据层：`Neo4j`
2. 服务层：`FastAPI`
3. 展示层：`React + TypeScript + react-force-graph-3d`

职责划分：

- `Neo4j` 负责图数据存储、图关系查询、全文搜索
- `FastAPI` 负责查询封装、参数校验、统一返回结构
- `React` 负责搜索交互、3D 渲染、详情面板

## 3. 为什么这样选

### 数据库

首选 `Neo4j`，不引入 MySQL。

原因：

- 当前目标是“图谱展示和搜索”，主查询天然是图查询
- 节点和关系模型直接落在 Neo4j 中最自然
- 避免双库同步，先降低系统复杂度

### 后端语言

首选 `Python`。

原因：

- `FastAPI` 适合快速搭建查询接口
- `neo4j` 官方 Python 驱动成熟
- 后续如果接入数据清洗、实体归并、LLM 能力，Python 扩展成本最低

### 前端语言

首选 `TypeScript`。

原因：

- 图节点和边的数据结构较复杂，类型约束很有价值
- React 生态中 3D 图谱库支持成熟
- 便于后期扩展筛选器、详情面板、配置面板

## 4. 业务模型

第一阶段只保留 5 类节点：

- `Person`
- `Band`
- `Work`
- `Album`
- `Genre`

第一阶段只保留 7 类关系：

- `(:Person)-[:PERFORMED]->(:Work)`
- `(:Person)-[:COMPOSED]->(:Work)`
- `(:Person)-[:WROTE_LYRICS_FOR]->(:Work)`
- `(:Person)-[:MEMBER_OF]->(:Band)`
- `(:Work)-[:IN_ALBUM]->(:Album)`
- `(:Work)-[:HAS_GENRE]->(:Genre)`
- `(:Band)-[:HAS_GENRE]->(:Genre)`

## 5. 功能范围

### 必做

- 关键词搜索实体
- 类型过滤搜索
- 查看节点详情
- 从某个节点展开 1 跳或 2 跳关系
- 查看两个实体之间的最短路径
- 3D 图谱交互展示

### 可以第二批再做

- 图谱截图导出
- 收藏视图
- 子图保存
- 时间过滤
- 热度排序

## 6. 查询流

### 搜索流

1. 用户输入关键词
2. 前端请求 `/search`
3. 后端在 Neo4j 中按 `name`、`aliases`、全文索引查询
4. 返回实体列表
5. 用户点击某条结果
6. 前端请求 `/graph/{id}?depth=1`
7. 后端返回中心节点和邻居关系
8. 前端渲染 3D 子图

### 展开流

1. 用户点击图中的某个节点
2. 前端再次请求 `/graph/{id}?depth=1`
3. 后端返回新的邻居子图
4. 前端进行去重合并
5. 重新布局并高亮新展开部分

### 路径流

1. 用户选择两个实体
2. 前端请求 `/path?from={id1}&to={id2}`
3. 后端查询最短路径
4. 前端高亮路径节点和边

## 7. 非功能要求

### 性能

- 单次默认返回的子图节点数建议限制在 `50`
- 默认展开深度最大限制为 `2`
- 搜索接口目标响应时间小于 `500ms`
- 子图查询目标响应时间小于 `800ms`

### 可维护性

- API 统一返回结构
- 节点类型和边类型集中定义
- 前后端共用数据结构文档

### 可扩展性

- 后续可以增加 `Label`、`Instrument`、`Place`
- 后续可以增加文本块和 GraphRAG
- 后续可以从单库升级为 `MySQL + Neo4j`

## 8. 项目阶段建议

### Phase 1

- 建 Neo4j schema
- 导入样例数据
- 完成搜索接口
- 完成节点详情接口
- 完成子图查询接口

### Phase 2

- 完成 React 页面
- 接入 3D 图谱渲染
- 支持点击展开与路径高亮

### Phase 3

- 优化布局
- 加入筛选、图例、详情面板
- 加入基础缓存

## 9. 结论

第一阶段最合适的组合是：

- 数据库：`Neo4j`
- 后端：`Python + FastAPI`
- 前端：`React + TypeScript`
- 图谱渲染：`react-force-graph-3d`

这套方案对“音乐图谱展示和搜索”是最直接、风险最低、后续扩展空间也足够大的组合。
